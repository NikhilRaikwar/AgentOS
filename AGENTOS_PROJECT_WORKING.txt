AgentOS Project Working Explanation
==================================

Project Name
------------
AgentOS

One-line Pitch
--------------
AgentOS is an operating system for onchain AI agents: users deploy wallet-owned agents with ENS names, machine-readable capability records, Uniswap-powered value movement, and KeeperHub execution reliability.


What The Project Does
---------------------
AgentOS lets a user connect a Sepolia wallet and create a real AI agent under agentos.eth.

Example:
  trade.agentos.eth
  research.agentos.eth
  analyst.agentos.eth

Each deployed agent gets:
  1. A smart wallet created through AgentWalletFactory.
  2. A real ENS subname under agentos.eth.
  3. ENS text records that describe the agent's capabilities.
  4. An ERC-8004 identity NFT.
  5. An AgentOS registry entry.
  6. OpenAI tool runtime for reasoning.
  7. Uniswap Trading API access for quote/swap/payment preparation.
  8. KeeperHub Direct Execution integration for reliable transaction execution.

The user owns the agent wallet. The server does not own user agents and does not need the user's private key.

--------------------------
The project has already produced a real Uniswap swap transaction on Sepolia:

ENS Integration
---------------
ENS is not cosmetic in this project. ENS is the identity, discovery, and metadata layer for agents.

What ENS does:
  - Each agent gets a real ENS subname under agentos.eth.
  - The subname resolves to the agent smart wallet.
  - ENS text records store machine-readable agent metadata.
  - Other agents and users can discover agents by resolving ENS names.
  - Agent capabilities are not stored only in a central database.

Example ENS text records:
  specialty=trading,defi,rebalancing
  fee=0.001 ETH
  preferred_token=USDC
  endpoint=http://localhost:3001/agents/name/run
  model=OpenAI
  reputation=50
  tasks_done=0
  framework=agentos/1.0
  keeperhub=enabled
  wallet_type=user-owned-smart-wallet

Why this matches ENS Best Integration for AI Agents:
  - AI agents need persistent, human-readable identities.
  - AgentOS gives every agent a name like name.agentos.eth.
  - ENS resolves the agent wallet address.
  - ENS stores capability metadata.
  - ENS enables agent discovery without hardcoded addresses.

Why this matches ENS Most Creative Use:
  - ENS text records become a machine-readable capability manifest.
  - Agents can inspect other agents through ENS records.
  - Subnames act as agent identity handles.
  - Reputation and task metadata can be written as ENS records.


Uniswap Integration
-------------------
Uniswap is the financial execution and settlement layer for AgentOS agents.

What Uniswap does:
  - The backend uses the Uniswap Trading API.
  - Agents request quotes before execution.
  - Agents can prepare swaps from quote responses.
  - Agents can route payments into another agent's preferred token.
  - The swap transaction is transparent and verifiable onchain.

Real proof:
  - A Sepolia ETH to Sepolia USDC swap was executed successfully.
  - The Uniswap quote endpoint returned a CLASSIC route.
  - The prepared transaction was broadcast and confirmed.

Uniswap API flow used:
  1. POST /quote
  2. POST /swap
  3. Validate transaction calldata
  4. Route execution through KeeperHub first
  5. Fallback direct executor broadcast only with explicit user approval

Why this matches Uniswap Best API Integration:
  - The agent actually uses the Uniswap Trading API.
  - The agent can quote token swaps.
  - The agent can prepare executable swap calldata.
  - The project demonstrates agentic finance instead of a normal swap UI.
  - The transaction is real and visible on Etherscan.


KeeperHub Integration
---------------------
KeeperHub is integrated as the reliability and execution layer for onchain agent transactions.

What KeeperHub does in AgentOS:
  - AgentOS prepares a transaction using the Uniswap API.
  - The backend sends the prepared transaction to KeeperHub Direct Execution.
  - KeeperHub is expected to handle reliable execution, retries, gas optimization, private routing, and audit trails.

Current KeeperHub result:
  - KeeperHub API authentication works.
  - KeeperHub /user endpoint returns the authenticated organization/user.
  - Direct execution currently fails because the KeeperHub dashboard has no execution wallet configured.

Required KeeperHub dashboard action:
  - Create/configure an execution wallet in the KeeperHub dashboard.
  - Fund it on Sepolia.
  - Make sure the organization API key has permission to use that wallet for Direct Execution.

Why this matches KeeperHub track:
  - KeeperHub is connected to a real agent execution workflow.
  - It is not only mentioned in the README.
  - The code calls KeeperHub before direct fallback.
  - The project discovered an actionable integration blocker: organization API key works, but wallet configuration is required.
  - This gives strong builder feedback for KeeperHub's feedback bounty.


OpenAI Agent Runtime
--------------------
AgentOS uses OpenAI for agent reasoning and tool-calling.

Tools exposed to the agent:
  - ENS discovery tool
  - Uniswap quote tool
  - KeeperHub history/execution tool

The agent is instructed:
  - Never claim execution success unless a tool result proves it.
  - Always show a Uniswap quote before value movement.
  - Require confirmation before broadcasting transactions.

User Flow
---------
1. User opens AgentOS.
2. User connects wallet on Sepolia.
3. User clicks Deploy Agent.
4. User enters:
     - agent name
     - specialty
     - fee
     - preferred token
5. User signs wallet transactions.
6. AgentOS creates:
     - smart wallet
     - ENS subname
     - ENS text records
     - ERC-8004 identity
     - registry entry
7. Dashboard shows only real deployed agents.
8. User selects the agent and asks it to quote or prepare a swap.
9. Agent uses OpenAI tools, ENS records, Uniswap API, and KeeperHub execution path.


How Other Users Can Discover Agents
-----------------------------------
Other users or agents can use an AgentOS agent if they know or search its ENS name.

Discovery path:
  1. Resolve name.agentos.eth.
  2. Read the wallet address.
  3. Read ENS text records.
  4. Check specialty, endpoint, fee, preferred token, and reputation.
  5. Call the endpoint or pay the agent using its preferred token.

How an orchestrator knows what each agent does:
  - It reads the specialty text record.
  - Example: specialty=research,analysis,defi-data means research agent.
  - Example: specialty=trading,defi,rebalancing means trading agent.
  - Example: specialty=orchestration,multi-agent means coordinator agent.


KeeperHub Dashboard Setup Needed
--------------------------------
To make KeeperHub execute the next transaction directly:

1. Open:
   https://app.keeperhub.com/

2. Log in with the same organization/account used to create the API key.

3. Go to Settings or Wallets.

4. Create or connect an execution wallet for the organization.

5. Select Sepolia as a supported network if the dashboard asks for network scope.

6. Fund that KeeperHub execution wallet with Sepolia ETH.

7. Make sure the organization API key has Direct Execution permission.

8. Re-run AgentOS backend /health:
   http://127.0.0.1:3001/health

9. Run another swap.

Expected behavior after wallet setup:
  - KeeperHub should accept /execute/contract-call.
  - KeeperHub should return an execution id or tx hash.
  - AgentOS should no longer need direct executor fallback.


Important Current Limitation
----------------------------
KeeperHub execution is integrated and authentication works, but direct KeeperHub transaction broadcast is blocked until the KeeperHub dashboard has an execution wallet configured.

This is not an AgentOS code failure. It is a KeeperHub organization setup requirement.
